# System Requirements

## Minimum Final Prototype Functionalities

These requirements define the minimum prototype functionalities that each ICT105 student group must address in the final prototype. The requirements are designed to be platform-independent and can align with different project topics, such as booking systems, reporting systems, inventory systems, learning platforms, dashboards, marketplace concepts, reminder systems, service request systems, or other IT-based solutions.

| Req ID | Minimum Prototype Functionality | What Students Must Show in Final Prototype |
| --- | --- | --- |
| **FR-01** | **Education Hub Homepage** | The prototype must clearly show the Education Hub title, target users (students), purpose of the platform, and main navigation options. |
| **FR-02** | **Primary user pathway** | The prototype must show how a student uses the system from start to finish. Example: Open Education Hub → Search Resource → View Details → Save/Register → Track Progress. |
| **FR-03** | **Resource or Event Submission** | The system must allow users to submit information related to the selected case. Example: report item, book appointment, register event, submit complaint, add inventory, request service. |
| **FR-04** | **Data Storage and Management** |Submitted resources and events must be stored and displayed using a database, spreadsheet, local storage, Firebase, or another suitable tool. |
| **FR-05** | **Resource and Event Listing** |Users must be able to view available resources, courses, workshops, tutorials, and academic events. |
| **FR-06** | **Search and Filter Function** | Users must be able to search by keyword and filter by category, subject, date, or type of resource. |
| **FR-07** | **Detailed Information View** |Users must be able to open a resource or event and view complete details, including description, category, organizer, and schedule. |
| **FR-08** | **Status Tracking** |The system must display statuses such as Upcoming, Registered, In Progress, Completed, or Expired.|
| **FR-09** | **Admin Management** | Admins must be able to approve, edit, update, or remove resources and events. |
| **FR-10** | **Input Validation** | The system must prevent incomplete submissions through required fields, valid email checks, and error messages.|
| **FR-11** | **Confirmation Messages** |Users must receive feedback after actions, such as "Registration Successful" or "Resource Submitted Successfully."|
| **FR-12** | **Dashboard and Analytics** | The prototype must display a summary dashboard showing total resources, total events, active users, and popular categories. |
| **FR-13** | **Consistent User Interface** |All screens must have consistent navigation, colors, buttons, labels, and layouts. |
| **FR-14** | **Responsive Design** |The prototype must work properly on both desktop and mobile devices. |
| **FR-15** | **Privacy and Responsible Data Handling** | The system should only collect necessary information and provide a basic privacy notice. |
| **FR-16** | **Prototype Traceability** |Every major screen and feature must be linked to the identified student needs, survey findings, and MVP requirements. |

## Minimum Technical Requirement

The final prototype must not be only a static idea or presentation. It must be an interactive prototype where users can click, submit, view, search/filter, update, or simulate the main workflow.

Acceptable platforms may include:

| Platform Type | Acceptable Prototype Form |
| --- | --- |
| Web app | HTML/CSS/JavaScript, PHP/MySQL, React, Laravel, or similar |
| Mobile app prototype | Figma clickable prototype, Flutter, React Native, MIT App Inventor, or similar |
| Dashboard system | Power BI, Tableau, Google Looker Studio, Excel dashboard, or web dashboard |
| Low-code/no-code system | Google Forms + Sheets, Airtable, Notion, Glide, AppSheet, Bubble, or similar |
| Data-driven prototype | Spreadsheet/database-backed prototype with forms, views, filters, and dashboard |
| UI/UX prototype | Figma prototype, but it must include realistic user flow, data screens, status screens, and interaction logic |
